# Build Command

You execute implementation plans generated by `/m:plan` or `/m:reverse-plan`. You parse the plan file, generate a tasks.json dispatch file, and launch the dispatch loop that executes tasks autonomously until all BDD tests pass.

**Arguments:** $ARGUMENTS

**All user interaction MUST use the AskUserQuestion tool.** Never ask questions as plain text in your response. This keeps you in control of the conversation flow.

## Step 1: Load Skills

Read skills that govern this command:

1. `${CLAUDE_PLUGIN_ROOT}/build/skills/dispatch/SKILL.md` — dispatch rules, tasks.json schema, status lifecycle
2. `${CLAUDE_PLUGIN_ROOT}/shared/skills/gherkin/SKILL.md` — BDD conventions, runner detection

Follow these skills' rules for all subsequent steps.

## Step 2: Parse Arguments

Parse `$ARGUMENTS` for:
- **Plan name** — the plan file name or stem (e.g., `202603261430-user-authentication` or `202603261430-user-authentication.md`)
- **`--resume` flag** — if present, resume a previously started build

Resolve the plan name to a full path at `.molcajete/plans/{name}.md` (append `.md` if missing).

If the plan file does not exist:
1. List available plans: `ls .molcajete/plans/*.md`
2. Report: "Plan file not found: {name}. Available plans: {list}"
3. Stop.

## Step 3: Validate Prerequisites

1. Verify the plan file exists and is readable.
2. Verify `bdd/` directory exists. If not: "BDD scaffold not found. Run `/m:scenario` first to generate Gherkin and step stubs." Then stop.
3. Verify step definition stubs exist in `bdd/steps/`. If empty: "No step definition stubs found in bdd/steps/. Run `/m:scenario` first." Then stop.

## Step 4: Read Plan File

Read the plan file and extract all task metadata. For each task section (`### T-NNN:`), extract:

| Field | Pattern |
|-------|---------|
| ID | `### T-NNN:` heading |
| Title | Text after `T-NNN:` in heading |
| Use Cases | `**Use Cases:**` line |
| Feature | `**Feature:**` line — extract the FEAT-XXXX ID |
| Domain | `**Domain:**` line |
| Architecture | `**Architecture:**` line |
| Intent | `**Intent:**` line (`implement` or `wire-bdd`) |
| Status | `**Status:**` line |
| Done signal | `**Done signal:**` line |
| Depends on | `**Depends on:**` line |

Parse the done signal to determine:
- **Signal type:** `bdd` if it references `@SC-XXXX` tags, `validator` otherwise
- **Tags:** Extract all `@SC-XXXX` tokens from the done signal text

Also read the plan header for:
- **Plan status** from `**Status:**`
- **Scope** from `**Scope:**`

## Step 5: Detect BDD Command

Determine the BDD runner command. Check in order:

1. `.molcajete/settings.json` — cached `bdd.framework` field
2. `bdd/CLAUDE.md` — runner hints
3. Step file extensions in `bdd/steps/` — `.py` = behave, `.ts`/`.js` = cucumber-js, `.go` = godog

Map to command:
| Framework | Command |
|-----------|---------|
| behave | `behave --no-capture` |
| cucumber-js | `npx cucumber-js` |
| godog | `godog` |

## Step 6: Handle Resume

If `--resume` flag is present OR `.molcajete/tasks.json` already exists:

1. Read `.molcajete/tasks.json`
2. Verify it references the same plan file. If not: "Existing tasks.json references a different plan ({other}). Delete it first or use the correct plan name." Then stop.
3. Show status summary via AskUserQuestion:

   For each task, show: ID, title, status.

   Count: implemented, failed, pending.

4. If any tasks have status `failed`, ask:
   - Question: "{count} task(s) failed in the previous run. Reset them to pending and retry?"
   - Header: "Resume Build"
   - Options: "Reset failed and resume" / "Resume as-is (skip failed)" / "Cancel"

   If "Reset failed": update tasks.json, set all `failed` tasks to `pending`, clear their errors.
   If "Resume as-is": continue without changes.
   If "Cancel": stop.

5. Skip to Step 9 (launch).

## Step 7: Generate tasks.json

Build the tasks.json file from the parsed plan data:

```json
{
  "plan_file": "{relative path to plan file}",
  "base_branch": "{current branch name}",
  "bdd_command": "{detected BDD command}",
  "tasks": [
    {
      "id": "T-001",
      "title": "{task title}",
      "feature": "FEAT-XXXX",
      "domain": "{domain}",
      "use_cases": ["UC-XXXX", "UC-YYYY"],
      "intent": "implement",
      "status": "pending",
      "done_signal": "bdd",
      "done_tags": ["@SC-XXXX", "@SC-YYYY"],
      "depends_on": [],
      "architecture": "prd/domains/{domain}/features/FEAT-XXXX-{slug}/ARCHITECTURE.md",
      "retries": 0,
      "commits": [],
      "error": null
    }
  ]
}
```

Get the current branch for `base_branch`:
```bash
git rev-parse --abbrev-ref HEAD
```

Write to `.molcajete/tasks.json`.

## Step 8: Confirm Launch

Show a task summary table via AskUserQuestion:

- Question: Format as a table with columns: Task ID, Title, Intent, Done Signal, Dependencies. Include totals: task count, BDD-gated count, validator-gated count.
- Header: "Build Plan: {plan name}"
- Options: "Launch" / "Cancel"

If "Cancel": stop. The tasks.json is preserved for later use with `--resume`.

## Step 9: Launch Dispatch

Run the dispatch loop:

```bash
bash "${CLAUDE_PLUGIN_ROOT}/build/scripts/dispatch.sh" ".molcajete/tasks.json"
```

This is a long-running process. Let it run to completion.

## Step 10: Post-Dispatch Report

After dispatch.sh exits, read the final state of `.molcajete/tasks.json` and report:

1. **Overall status:** All implemented, partially failed, or fully failed
2. **Per-task results:** ID, title, final status, error (if any)
3. **Commits:** Total commits across all tasks
4. **Plan file status:** Current plan-level status

If all tasks implemented:
- "Build complete. All {N} tasks implemented and merged to {branch}."

If some tasks failed:
- "Build partially complete. {done}/{total} tasks implemented. {failed} task(s) failed."
- List failed tasks with their errors.
- "Run `/m:build {plan-name} --resume` to retry failed tasks."

If all tasks failed:
- "Build failed. No tasks completed successfully."
- List all errors.

## Rules

- The plan file is the human-readable source of truth. tasks.json is the machine-readable dispatch state.
- dispatch.sh is the only process that modifies tasks.json during execution. Do not modify it while dispatch is running.
- Use AskUserQuestion for all user interaction — never ask as plain text.
- The `--resume` flow must handle tasks.json from a previous incomplete run gracefully.
- Never delete or overwrite an existing tasks.json without user confirmation.
