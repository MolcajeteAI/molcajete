# Build Command

You execute a single task from an implementation plan generated by `/m:plan` or `/m:reverse-plan`. This is the interactive entry point for single-task execution. For automated multi-task builds, use `molcajete build <plan-name>` which orchestrates the full session-based pipeline.

**Arguments:** $ARGUMENTS

**All user interaction MUST use the AskUserQuestion tool.** Never ask questions as plain text in your response. This keeps you in control of the conversation flow.

## Step 1: Load Skills

Read skills that govern this command:

1. `${CLAUDE_PLUGIN_ROOT}/build/skills/dispatch/SKILL.md` — dispatch rules, session types, dev-validate cycle, plan JSON schema
2. `${CLAUDE_PLUGIN_ROOT}/shared/skills/gherkin/SKILL.md` — BDD conventions, runner detection
3. `${CLAUDE_PLUGIN_ROOT}/shared/skills/git-committing/SKILL.md` — commit message standards

Follow these skills' rules for all subsequent steps.

## Step 2: Parse Arguments + Load Plan

Parse `$ARGUMENTS` — split into two values:
- **Plan name** — the plan file name or stem (e.g., `202603261430-user-authentication`)
- **Task ID** — the task to execute (e.g., `T-001`)

If either is missing, report usage: `/m:build <plan-name> <task-id>` and stop.

Resolve the plan name to a full path at `.molcajete/plans/{name}.json` (append `.json` if missing).

If the plan file does not exist:
1. List available plans: `ls .molcajete/plans/*.json`
2. Report: "Plan file not found: {name}. Available plans: {list}"
3. Stop.

Read the plan JSON, find the task object with matching `id` field. If `bdd_command` is null, detect per the **BDD Command Detection** section in the dispatch skill and write it back to the plan file.

Check dependencies: all tasks in `depends_on` must have status `implemented`. If not, report which dependencies are blocking and stop.

Set the task status to `in_progress` in the plan file and commit:
```bash
git add .molcajete/plans/{plan_file}
git commit -m "plan: mark {T-NNN} in_progress"
```

## Step 3: Load Project Context

Read these files (skip any that don't exist):

**Project-level:**
- `prd/PROJECT.md`
- `prd/TECH-STACK.md`
- `prd/DOMAINS.md`
- `CLAUDE.md` and all `.claude/rules/*.md`
- `.molcajete/apps.md` — project environment and tooling configuration (services, health checks, runtime, BDD, tooling commands, verification profiles). Generated by `/m:setup`. **If missing or has no Tooling section, warn the user:** "No tooling settings detected. Run `/m:setup` and choose 'Update tooling only' to configure."

**Commit style detection:**
```bash
git log --oneline -20
```

**Feature-level:**
- Feature's `REQUIREMENTS.md`: glob `prd/domains/{domain}/features/{feature}-*/REQUIREMENTS.md`
- Feature's `ARCHITECTURE.md`: read the task's `architecture` field directly

**Use case and Gherkin files:**
For each UC-XXXX in the task's Use Cases:
1. Find the UC file: glob `prd/domains/*/features/*/use-cases/{UC-XXXX}-*.md`
2. Read the UC file
3. Find the Gherkin feature file: grep `bdd/features/` for `@{UC-XXXX}`
4. Read the feature file

**Step definitions index:**
- `bdd/steps/INDEX.md`

**Prior task summaries:**
- Read summaries from completed prior tasks in the plan file

## Step 4: Create Worktree

Per the dispatch skill's **Worktree Management** section — create a worktree for this task from the plan's base branch. Handle stale worktrees if present.

All subsequent work happens in this worktree path.

## Step 5: Implement

Per the dispatch skill's **Implementation** sections — execute the procedure matching the task's intent:

- `implement` → Phase A (production code + unit tests) then Phase B (step definitions + docs)
- `wire-bdd` → Single phase (step definitions + docs)

If the task has sub-tasks, execute them sequentially in the worktree. Each sub-task follows the same implementation procedure. After all sub-tasks complete, run a task-level validation (including BDD).

## Step 6: Quality Gates

Run all quality gates per the dispatch skill's **Quality Gates** section.

In interactive mode, you run the gates yourself rather than delegating to the validation session:

**All gates in parallel (read-only checks):**
- Formatting: run formatter in check mode
- Linting: run linter without --fix
- BDD tests: run with done_tags filter
- Code review: self-review for intent conformance
- Completeness: trace requirements to code

**Batch tooling lookup:** Read `.molcajete/apps.md` → Tooling section based on intent.

If all gates pass → proceed to Step 7.

If any gate fails → fix ALL issues in one patch, commit, re-run all gates. Max 7 attempts.

After exhausting retries → set task status to `failed`, skip to Step 9.

## Step 7: Final Validation

Run ALL project tests — full BDD suite without tag filter, full unit tests. Every task must pass this step.

**Setup errors are a hard stop.** If tests cannot run due to infrastructure issues, stop immediately — set task to `failed` and skip to Step 9.

**If tests fail due to code issues:** Fix the code, commit, re-run. Do NOT proceed with failing tests.

## Step 8: Merge

Per the dispatch skill's **Merge Procedure** section — rebase onto base branch, fast-forward merge, clean up worktree.

If conflicts occur, resolve them per the **Conflict Resolution** section.

## Step 9: Finalize

### On success (merge completed)

The plan file update and commit happened in Step 8.

### On failure

- Set status to `failed` and write `error` in the plan JSON
- Commit the plan file update

### Report result
Report clearly to the user:
- Task status (done/failed)
- Commits made (list SHAs)
- Quality gate results
- Summary + watch-outs
- Next task suggestion (if other pending tasks exist)

## Rules

- The plan JSON file is the single source of truth.
- Use AskUserQuestion for all user interaction — never ask as plain text.
- For `implement` intent: two commits — production code (Phase A), then step definitions + docs (Phase B).
- For `wire-bdd` intent: one commit — step definitions + docs. Never modify production code.
- Docs are always folded into the final commit — never a separate docs commit.
- Formatter and linter must run sequentially (chained command), never in parallel — both modify files.
- Fix ALL quality gate issues in one patch before re-running gates, never one at a time.
- **MANDATORY: ALL tests must pass.** A task is NEVER `done` if any test is failing. BDD tests must pass 100%.
- **The word SKIP does not exist in the vocabulary of test execution.**
