# Build Command

You execute a single task from an implementation plan generated by `/m:plan` or `/m:reverse-plan`. One session handles everything: worktree creation, implementation, quality gates, merge, and cleanup.

**Arguments:** $ARGUMENTS

**All user interaction MUST use the AskUserQuestion tool.** Never ask questions as plain text in your response. This keeps you in control of the conversation flow.

## Step 1: Load Skills

Read skills that govern this command:

1. `${CLAUDE_PLUGIN_ROOT}/build/skills/dispatch/SKILL.md` — dispatch rules, plan JSON schema, status lifecycle, all procedures
2. `${CLAUDE_PLUGIN_ROOT}/shared/skills/gherkin/SKILL.md` — BDD conventions, runner detection
3. `${CLAUDE_PLUGIN_ROOT}/shared/skills/git-committing/SKILL.md` — commit message standards, atomic commits

Follow these skills' rules for all subsequent steps.

## Step 2: Parse Arguments + Load Plan

Parse `$ARGUMENTS` — split into two values:
- **Plan name** — the plan file name or stem (e.g., `202603261430-user-authentication`)
- **Task ID** — the task to execute (e.g., `T-001`)

If either is missing, report usage: `/m:build <plan-name> <task-id>` and stop.

Resolve the plan name to a full path at `.molcajete/plans/{name}.json` (append `.json` if missing).

If the plan file does not exist:
1. List available plans: `ls .molcajete/plans/*.json`
2. Report: "Plan file not found: {name}. Available plans: {list}"
3. Stop.

The plan file IS the JSON state. Read `base_branch` and `bdd_command` from top-level fields. If `bdd_command` is null, detect per the **BDD Command Detection** section in the dispatch skill and write it back to the plan file.

Read the plan JSON, find the task object with matching `id` field. Extract:
- `title`, `use_cases`, `feature`, `domain`, `architecture`, `intent`, `done_signal`, `done_tags`, `depends_on`

Check dependencies: all tasks in `depends_on` must have status `implemented` in the plan file. If not, report which dependencies are blocking and stop.

Set the task status to `in_progress` in the plan file.

## Step 3: Load Project Context

Read these files (skip any that don't exist):

**Project-level:**
- `prd/PROJECT.md`
- `prd/TECH-STACK.md`
- `prd/DOMAINS.md`
- `CLAUDE.md` and all `.claude/rules/*.md`
- `.molcajete/settings.json` — contains tooling commands (format, lint, test per domain), environment config (Docker, start/stop), BDD framework, and available scripts. **If this file is missing or has no `tooling` key, warn the user:** "No tooling settings detected. Run `/m:setup` and choose 'Update tooling only' to configure formatter, linter, and test commands. The build will attempt to detect tools on-the-fly, but this is less reliable."

**Commit style detection:**
```bash
git log --oneline -20
```

**Feature-level:**
- Feature's `REQUIREMENTS.md`: glob `prd/domains/{domain}/features/{feature}-*/REQUIREMENTS.md` (using the task's `domain` and `feature` fields)
- Feature's `ARCHITECTURE.md`: read the task's `architecture` field directly (full path with slug, resolved at plan time)

**Use case and Gherkin files:**
For each UC-XXXX in the task's Use Cases:
1. Find the UC file: glob `prd/domains/*/features/*/use-cases/{UC-XXXX}-*.md`
2. Read the UC file
3. Find the Gherkin feature file: grep `bdd/features/` for `@{UC-XXXX}`
4. Read the feature file

**Step definitions index:**
- `bdd/steps/INDEX.md`

**Prior task summaries:**
- Read summaries from completed prior tasks in the plan file (`summary` fields)

## Step 4: Create Worktree

Per the dispatch skill's **Worktree Management** section — create a worktree for this task from the plan's base branch. Handle stale worktrees if present.

All subsequent work happens in this worktree path.

## Step 5: Implement

Per the dispatch skill's **Implementation** sections — execute the procedure matching the task's intent:

- `implement` → **Implementation: `implement` Intent** (Phase A + Phase B)
- `wire-bdd` → **Implementation: `wire-bdd` Intent** (single phase)

## Step 6: Quality Gate Loop (max 5 attempts)

Per the dispatch skill's **Quality Gate (Self-Validation)** section — run all gates in two sequential batches, fix all failures in one patch, retry up to 5 times.

**Batch 1 tooling lookup:** Read `.molcajete/settings.json` → `tooling` section. For `wire-bdd` tasks, use `tooling.bdd` (format + lint commands for step definitions). For `implement` tasks, use both `tooling.{domain}` (production code) and `tooling.bdd` (step definitions). Never skip formatting or linting when a tooling entry exists — run the commands.

If all gates pass → proceed to Step 7.

After 5 failures → set task status to `failed`, record the error, skip to Step 9.

## Step 7: Final Validation

Run ALL project tests (full BDD suite without tag filter, full unit tests).

**Setup errors are a hard stop.** If tests cannot run due to infrastructure issues (services down, missing dependencies, database unreachable), stop immediately — set task to `failed` with `"Setup error: {description}"` and skip to Step 9. Do not retry, do not attempt to fix infrastructure.

If tests run but fail due to code issues caused by this task's changes, fix them and commit.

If failures are pre-existing (existed before this task), note them in watch_outs but do not fail.

## Step 8: Merge

Per the dispatch skill's **Merge Procedure** section — rebase onto base branch, fast-forward merge, clean up worktree.

If conflicts occur, follow the **Conflict Resolution** section.

If merge fails completely, set task status to `failed` and skip to Step 9.

## Step 9: Finalize

### Update task status
- Set status in plan file: `implemented` (success) or `failed`
- Set the task's `summary` field (for successful tasks, per dispatch skill's **Summary Writing** section)

### Report result
Report clearly to the user:
- Task status (done/failed)
- Commits made (list SHAs)
- Quality gate results (pass/fail/skip for each)
- Summary + watch-outs
- Next task suggestion (if other pending tasks exist in the plan)

Note: Architecture update, PRD status propagation, and README updates are folded into the task's final commit (Step 5), not done separately here. See the dispatch skill's **Docs Update (Pre-Commit)** section.

## Rules

- The plan JSON file is the single source of truth for both plan content and dispatch state.
- Use AskUserQuestion for all user interaction — never ask as plain text.
- For `implement` intent: two commits — production code (Phase A), then step definitions + docs (Phase B).
- For `wire-bdd` intent: one commit — step definitions + docs. Never modify production code.
- Docs (ARCHITECTURE.md, PRD status, README.md) are always folded into the final commit — never a separate docs commit.
- For `implement` intent: update/create README.md files in code directories that were created or significantly modified.
- Formatter and linter must run sequentially (chained command), never in parallel — both modify files.
- Fix ALL quality gate issues in one patch before re-running gates, never one at a time.
- When running quality gates, Batch 2 gates (BDD, code review, completeness) should all be issued as tool calls in a single turn.
- The task agent handles everything: worktree, implementation, quality gates, merge, cleanup. No separate review or architecture agents.
- **MANDATORY: ALL tests must pass.** A task is NEVER `done` if any test is failing. BDD tests must pass 100% — no exceptions, no "pass with watch-outs", no skipping failures. If tests fail, fix the code. If retries are exhausted, the task is `failed` and the build STOPS. No subsequent tasks may start.
